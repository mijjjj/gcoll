# 开发任务规范

## 通用工作流

1. 先读 `docs/README.md` 和本任务相关的 `docs/current/` 文件。
2. 判断任务属于产品、架构、后端、前端、插件、部署或文档。
3. 修改公开契约前，先判断是否影响 HTTP API、插件协议、SDK、事件格式或数据库迁移。
4. MVP 内部契约采用最新设计优先，不添加未发布旧设计的兼容分支。
5. 完成实现后更新对应文档和测试。

## 后端开发规则

- 使用 Go + GoFrame。
- 后端目录采用 GoFrame 三层结构：`api/<domain>/v1`、`internal/controller/<domain>`、`internal/service/<domain>`，运行时装配放在 `internal/cmd`。
- HTTP 接口优先使用 GoFrame 标准路由和 `group.Bind(...)`，避免继续扩散手写 `ghttp.Request` 风格控制器。
- 管理 API 默认返回 `code`、`message`、`data` 统一包裹结构，由中间件处理，不在控制器里重复封装。
- 模块边界遵循 `docs/current/02-系统架构.md`。
- 控制面不得进入高频数据路径。
- 数据面不得依赖 HTTP 请求生命周期。
- 高频采集路径不得逐条同步写库。
- 数据库迁移必须显式版本化。
- 同一迁移无法同时兼容 SQLite 和 PostgreSQL 时，迁移文件必须按 `manifest/migrations/sqlite/` 与 `manifest/migrations/pgsql/` 分目录维护，文件名版本号保持一致。
- 迁移文件版本前缀统一使用 `000001_变更说明` 这种六位顺序号格式，便于按执行顺序查看和排查。
- 结构体、接口、函数、字段注释必须使用中文。
- 日志字段、配置键、协议字段保持约定英文。

## 插件开发规则

- 插件默认独立进程运行。
- 插件通信只使用 gRPC。
- 插件配置由宿主保存，不得保存在插件目录作为最终运行配置。
- 南向插件只提交采集记录，不直接转发目标系统。
- 北向插件只接收过滤后的记录，不主动采集设备。
- 插件不得直接访问宿主数据库。
- 敏感值必须通过宿主密钥存储引用。
- 插件协议变更必须同步更新 `docs/current/03-插件与点位协议.md`。
- 具体南向协议的连接字段、点位 metadata、读取计划、写入限制和调试摘要必须留在插件实现内，主程序不得引入协议专用表或协议专用 service 分支。

## 前端开发规则

- 使用 Vue 3 + TypeScript + Vite + Naive UI。
- 页面风格遵循 `docs/current/04-前端设计规范.md`。
- 服务端控制台和桌面端共享设计语言，桌面端更强调本机状态和调试。
- 表格、分栏、抽屉和表单优先，避免卡片墙。
- 插件自定义页面外层必须保留宿主工具栏、保存状态、权限提示和审计提示。
- 页面实现后必须检查浅色模式、黑夜模式和 `1280x800` 视口。

## 数据库和迁移规则

- 配置、任务、插件、用户、审计和必要运行状态可以落库。
- 采集明细数据不落库。
- 最新点位值保存在运行时缓存。
- 设备插件配置、点位表、插件配置结构必须支持版本化。
- 敏感配置不进入普通 JSON 字段明文。

## 测试要求

不同任务至少覆盖对应风险：

- 插件清单验证、签名验证、校验和验证。
- 插件安装、启用、启动、停止、升级、回滚。
- 插件握手、协议版本兼容和健康检查。
- 不同设备使用同一插件时配置隔离。
- 默认配置表单和服务端最终校验。
- 敏感字段不明文返回。
- 通用点位表增删改查、导入导出和校验。
- 最新点位缓存更新和变化上报。
- 规则过滤后再转发给北向插件。
- 前端页面浅色、黑夜、空状态、错误状态和只读权限状态。

## 文档维护规则

- 当前有效设计只写入 `docs/current/`。
- ADR 只记录已确认且难以逆转的决策。
- 不要为同一主题新增多个平级草稿文档。
- 文档和代码注释必须使用中文。
- 机器可读字段、命令、路径、API 路径、配置键和协议字段保持英文。

## 交付检查清单

完成任务前检查：

- 是否读过相关 `docs/current/` 文件。
- 是否遵守模块边界。
- 是否影响公开契约。
- 是否补充或更新测试。
- 是否更新相关文档。
- 是否存在中文注释和中文说明缺失。
- 是否避免了旧内部契约兼容分支。
