# ADR-0004 数据面采用内存缓存、规则过滤和背压机制

## 状态

建议采纳。

## 背景

数据采集和转发是 gcoll 的核心性能路径。南向插件可能产生高频数据，北向目标系统行为差异较大。当前产品不需要保存采集历史数据，因此数据面应优先围绕最新点位缓存、规则过滤和插件转发性能设计。

## 决策

数据面采用内存缓存和规则过滤管道：

- 南向插件批量提交采集记录。
- 宿主写入分区内存缓冲并更新最新点位缓存。
- 采集数据不落库。
- 插件配置支持立即上报和变化上报，默认变化上报。
- 数据必须先经过规则过滤，再转发给北向插件。
- 框架只负责将过滤后的消息转发给北向插件。
- 北向目标系统的投递、重试、确认和幂等由北向插件自行决定。
- 数据面不引入外部队列。

## 影响

正面影响：

- 提高吞吐。
- 降低数据库写入压力。
- 最新点位读取快。
- 北向投递语义更灵活，由插件按目标系统自行实现。
- 更容易做性能压测和容量规划。

负面影响：

- 系统复杂度增加。
- 不能查询历史采集明细数据。
- 进程重启后最新点位缓存需要重新采集恢复。
- 框架不承诺目标系统投递成功。

## 性能策略

- 按设备、任务或路由进行分区。
- 批量大小可配置。
- 工作池大小可配置。
- 支持最新点位缓存容量限制。
- 支持立即上报和变化上报。
- 支持规则过滤。
- 指标和日志异步聚合。

## 备选方案

- 同步采集转发链路：实现简单，但吞吐和抗抖动能力弱。
- 首版直接引入 Kafka、NATS 或 Redis Streams：扩展能力强，但与当前“不引入外部队列”的要求不符。
- 保存全量采集历史数据：便于追溯，但会显著增加数据库和磁盘压力。

## 后续动作

- 实现最新点位缓存。
- 定义背压状态。
- 定义规则过滤接口。
- 定义北向插件转发接口。
- 通过压测确定默认批量大小和工作池大小。
