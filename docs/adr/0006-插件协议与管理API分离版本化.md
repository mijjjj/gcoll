# ADR-0006 插件协议与管理 API 分离版本化

## 状态

建议采纳。

## 背景

管理 API 面向前端、CLI 和第三方系统集成；插件协议面向插件进程和 SDK。二者变更频率、性能要求、调用方式和兼容策略不同。如果共用同一套版本，会导致不必要的兼容压力。

## 决策

插件协议和管理 API 分离版本化：

- 管理 API 使用 `/api/v1` 路径版本。
- 插件协议使用独立 `pluginProtocol` 版本。
- 插件通信协议只支持 gRPC。
- 其他语言插件通过 gRPC 生成代码接入。
- 插件清单声明兼容的核心运行时版本和插件协议版本。
- 宿主启动插件前必须检查协议版本兼容性。

## 影响

正面影响：

- 插件生态版本边界清晰。
- 管理 API 可以独立演进。
- 宿主可以明确拒绝不兼容插件。
- SDK 和协议测试范围更集中。

负面影响：

- 需要维护两套版本策略。
- 文档和测试需要分别覆盖。

## 备选方案

- 管理 API 和插件协议共用版本：简单但耦合过高。
- 插件不声明协议版本：短期省事，长期兼容风险大。
- 同时支持 JSON-RPC：生态更宽，但当前阶段会增加协议、测试和宿主复杂度。

## 后续动作

- 在插件清单中保留 `compatibility.pluginProtocol`。
- 在宿主握手时检查协议版本。
- 在 SDK 中暴露协议版本常量。
