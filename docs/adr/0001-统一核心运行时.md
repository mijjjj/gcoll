# ADR-0001 统一核心运行时支撑服务器端和桌面端

## 状态

建议采纳。

## 背景

gcoll 需要同时支持服务器部署和桌面端部署。服务器端适合集中化、长期运行和企业内网部署；桌面端适合现场调试、边缘采集和轻量化运行。如果两种形态分别实现业务逻辑，会导致插件协议、采集任务、配置模型和转发逻辑重复建设。

## 决策

采用统一核心运行时：

- `gcoll-server` 和 `gcoll-desktop` 共用核心业务模块。
- 桌面端通过 Wails 3 作为外壳，调用同一套运行时服务。
- 服务器端通过 GoFrame HTTP API 暴露控制面能力。
- 数据面不依赖具体入口形态。

## 影响

正面影响：

- 降低维护成本。
- 插件只需要适配一套运行时协议。
- 桌面端和服务器端行为更一致。
- 测试可以复用。

负面影响：

- 核心运行时需要从一开始就区分运行模式。
- 配置、数据库和文件路径需要抽象。
- 桌面端能力和服务器端能力的边界需要清晰管理。

## 备选方案

- 服务器端和桌面端分别实现：短期开发直观，但长期重复严重。
- 桌面端只作为远程服务器客户端：无法满足离线采集和本地部署需求。

## 后续动作

- 在代码结构中建立 `internal/runtime` 或等价核心启动层。
- 为 server、desktop、cli 提供不同入口，但复用同一套服务注册。
- 在配置中明确 `runtime.mode`。
