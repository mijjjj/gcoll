# ADR-0011 通用点位表作为默认点位模型

## 状态

已采纳。

## 背景

所有设备都需要配置采集点位。不同协议的点位地址语义不同，但大多数点位都可以抽象为名称、描述、地址和值类型等通用字段。如果每个插件都自行设计点位结构，会导致前端、缓存、规则过滤和导入导出能力难以复用。

## 决策

所有设备默认使用通用点位表。

通用点位表至少包含：

- 点位名称。
- 点位描述。
- 点位地址。
- 值类型。
- 单位。
- 是否启用。
- 标签。
- 插件扩展元数据。

当部分插件的点位模型无法使用通用点位表表达时，插件可以提供自带点位页面，并嵌入设备点位配置区域。

## 边界

- 设备连接参数、认证信息、采集周期、上报模式、调试模式等都在设备配置页维护。
- 通用点位表只描述采集点位。
- 插件自带点位页面只负责特殊点位配置，不替代设备配置页。
- 插件自带点位页面必须通过宿主点位 API 保存数据。

## 影响

正面影响：

- 点位管理体验统一。
- 最新点位缓存、变化上报和规则过滤可以基于统一点位 ID 实现。
- 导入导出和批量编辑更容易实现。

负面影响：

- 极少数复杂协议需要额外点位页面。
- 插件需要把协议点位映射到通用点位结构或自带点位页面。

## 后续动作

- 实现 `device_points` 表。
- 实现设备点位 API。
- 实现点位导入导出。
- 实现插件自带点位页面嵌入能力。
