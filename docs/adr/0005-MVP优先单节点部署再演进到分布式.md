# ADR-0005 MVP 优先单节点部署再演进到分布式

## 状态

建议采纳。

## 背景

gcoll 最终可能需要支持中心服务器、边缘节点和高吞吐采集工作节点。但 MVP 阶段的首要目标是验证本地插件体系、设备配置、采集缓存、规则过滤和北向插件转发链路。如果过早引入分布式调度、外部中间件或在线商城，会拉长闭环验证周期。

## 决策

MVP 优先支持：

- 单节点服务器部署。
- 单机桌面部署。
- 内置运行时队列。
- 本地插件进程。
- 本地插件安装和管理。
- PostgreSQL 或 SQLite。

后续再扩展：

- 中心服务器 + 边缘桌面。
- 多数据面工作节点。
- 分布式任务调度。
- 在线应用商城。

## 影响

正面影响：

- MVP 范围清晰。
- 部署简单。
- 便于快速验证完整链路。
- 后续可基于真实压测结果演进。

负面影响：

- 首版横向扩展能力有限。
- 高吞吐场景需要后续增加工作节点和更完整的调度能力。

## 备选方案

- 首版直接做分布式架构：能力完整，但复杂度和交付风险高。
- 只做桌面端：现场验证快，但不利于企业服务器部署。

## 后续动作

- 在任务模型中预留 `node_id` 或执行节点字段。
- 在运行时状态中记录节点心跳。
- 在插件系统中预留应用商城扩展接口。
