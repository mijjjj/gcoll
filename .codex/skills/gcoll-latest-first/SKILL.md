# gcoll Latest First

## 核心原则

1. 先判断本次修改是否只是内部实现调整；如果不是公开给第三方的稳定契约，默认不为旧实现保留兼容层。
2. 先满足当前最新需求，再考虑旧代码是否值得保留；如果旧设计已经阻碍实现、性能或边界清晰度，优先删除或重构旧设计。
3. 架构、接口、数据结构和读取写入路径优先选择更清晰、更短链路、更少状态复制的方案。
4. 性能关键路径优先减少无效转换、重复查询、重复序列化和控制面干预。

## 执行顺序

1. 先识别哪些内容属于公开契约：
   - HTTP API
   - 插件协议
   - SDK
   - 事件格式
   - 已经明确对第三方开放的配置或数据格式
2. 如果属于公开契约且存在破坏性变更风险，先向用户确认是否需要兼容策略。
3. 如果不属于公开契约，直接按最新设计修改，不额外保留旧分支、旧字段映射、旧参数格式或旧流程兜底。
4. 发现旧表结构、旧接口、旧状态机或旧页面结构明显妨碍实现时，可以连带清理或替换，但要同步更新当前文档与调用点。

## 默认做法

- 默认不保留“新旧两套逻辑并存”。
- 默认不增加仅为兼容旧内部代码而存在的适配层。
- 默认不保留废弃字段、废弃 metadata、废弃参数格式。
- 默认不为了迁就旧结构而牺牲批量处理、排序优化、缓存利用或模块边界。
- 默认允许重命名、删字段、删分支、删旧流程，只要当前需求和当前文档已同步收敛。

## 禁止事项

- 不要在未获用户明确要求的情况下添加兼容旧版本的隐式 fallback。
- 不要为了“可能有人还在用”而保留仓库内部未声明稳定性的旧接口。
- 不要把过渡期脏逻辑长期留在主流程。
- 不要把协议专属知识继续散落到宿主多个模块；能收回插件内部的，应优先收回插件内部。

## 交付要求

- 明确说明本次是否放弃旧兼容。
- 如果替换了旧架构、旧字段或旧接口，说明新的唯一合法格式或唯一有效路径。
- 完成后运行对应验证：
  - 后端改动运行 `go test ./...`
  - 前端改动运行 `pnpm --dir frontend/web build`
