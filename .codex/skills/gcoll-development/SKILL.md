# gcoll Development

## 使用流程

1. 先阅读 `docs/README.md`。
2. 按任务类型阅读 `docs/current/` 中的对应规范。
3. 判断是否影响公开契约：HTTP API、插件协议、SDK、事件格式、数据库迁移。
4. 内部实现按最新设计推进，不为未发布旧设计添加兼容分支。
5. 实现后补测试，并更新对应文档。

## 规范入口

- 产品范围：`docs/current/01-产品范围.md`
- 系统架构：`docs/current/02-系统架构.md`
- 插件与点位协议：`docs/current/03-插件与点位协议.md`
- 前端设计规范：`docs/current/04-前端设计规范.md`
- 开发任务规范：`docs/current/05-开发任务规范.md`
- 决策背景：`docs/adr/README.md` 和对应 ADR

## 强制约束

- 文档和代码注释使用中文。
- 机器字段、命令、路径、API 路径、配置键和协议字段保持英文。
- 控制面不得进入高频数据路径。
- 高频采集路径不得逐条同步写库。
- 插件不得直接访问宿主数据库。
- 南向插件只提交采集记录。
- 北向插件只接收过滤后的记录。
- 设备配置和点位表必须分离。
- 敏感配置必须通过宿主密钥存储引用。

## AI 执行规则

1. 先识别任务类型：产品、架构、后端、前端、插件、数据库、部署或文档。
2. 若影响 HTTP API、插件协议、SDK、事件格式或数据库迁移，先更新当前规范并说明版本策略或兼容选择。
3. 若只影响内部实现，按最新设计直接推进，不保留旧内部模式。
4. 修改代码前先搜索现有模式，优先复用项目结构。
5. 新增文档和代码注释使用中文。
6. 实现后补充最小但有效的测试。

## 前端检查

前端任务完成后检查：

- 浅色模式。
- 黑夜模式。
- `1280x800` 视口。
- 空状态。
- 错误状态。
- 只读权限状态。
- 插件自定义页面的主题、权限和保存状态。

## 本地验证

- 修改后端 Go 代码后，优先运行 `go test ./...`。
- 修改前端代码后，优先运行 `pnpm --dir frontend/web build`。
- 除非用户明确要求，不默认启动 `gcoll-server`、Vite 开发服务、前端预览服务或桌面端运行时。
- 除非用户明确要求，不默认打开浏览器验证本地页面。
- 如果用户明确要求启动服务或打开页面，验证完成后关闭本次启动的进程。
